# 🚨 СРОЧНОЕ ИСПРАВЛЕНИЕ: Доступ запрещён после включения RLS

## Что случилось?

Политики RLS блокировали чтение записи `employees` при входе в систему. Функция `fetchEmployeeForUser()` не смогла прочитать вашу запись → вход заблокирован.

---

## ⚡ ШАГ 1: СРОЧНО! Отключите RLS (прямо сейчас!)

### Откройте Supabase SQL Editor и выполните:

```sql
ALTER TABLE employees DISABLE ROW LEVEL SECURITY;
```

Или используйте файл: **`SQL_DISABLE_RLS_URGENT.sql`**

✅ **После этого вы сможете войти снова!**

---

## ⚡ ШАГ 2: Проверьте вход

1. Обновите страницу приложения (F5)
2. Попробуйте войти снова
3. Должно работать!

---

## ✅ ШАГ 3: Включите RLS правильно

Я создал **исправленные политики**, которые НЕ блокируют вход.

### Выполните файл: **`SQL_RLS_FIXED.sql`**

В нём правильные политики:

1. **"Enable read access for own record"**
   - Пользователь может читать свою запись
   - **Это нужно для входа в систему!**

2. **"Owners can read all active employees"**
   - Владельцы видят всех сотрудников
   - Нужно для управления пользователями

3. **"Users can update own telegram link token"**
   - Пользователи могут обновлять свой `link_token`
   - Нужно для привязки Telegram

## ❓ Что было не так?

Старые политики проверяли `auth.uid() = auth_user_id` для SELECT, но:
- При логине Supabase Auth ещё не полностью инициализирован
- `fetchEmployeeForUser()` не смог прочитать запись
- Система решила, что доступ запрещён

**Исправление:** Новые политики используют правильную логику для чтения записей при логине.
